Environment Promotion
Promote test cases from DEV to higher environments as deterministic replay only
P0 · 5–7 days
The Problem

Today, all environments use the same AI-powered authoring flow. In higher environments (QA, UAT, PROD), we need deterministic, repeatable execution — not AI re-interpretation. A test that passed in DEV should execute the exact same steps in QA.

→

UATdeterministic replay
→
PRODdeterministic replay
Design Decisions
DEV = Full AI Mode

AI can author, modify, and iterate on test plans. Test cases can be edited freely. This is the experimentation environment.

QA / UAT / PROD = Replay Only

Promoted test cases are locked. No AI re-generation. Only deterministic replay using the exact saved steps. UI hides the "Author Agent" chat in these environments.

Implementation SHIPPED
1. {service.KEY} Config References in Plan Steps Done

Agent generates plan steps with {service.BESTBUY_USA_URL} template references instead of hardcoded URLs. The LLM's response is post-processed: config values are replaced with {service.KEY} templates automatically. UI shows resolved values with the template in small gray text below.

2. promote_test_case Lambda Action Done

New action in p2t-testcase-writer Lambda. Copies test case to target env (ID: TC-xxx-qa), strips action_resolved/expected_resolved fields (templates only), promotes service config keys to target env DynamoDB with empty values, and updates source test case's promoted_to array.

3. Sequential Promotion (DEV → QA → UAT → PROD) Done

Promotion is sequential only: DEV can promote to QA, QA to UAT, UAT to PROD. No skipping environments. PROD has no promote option.

4. Promotion Pipeline Badge on Inventory UI Done

Each test case row shows an inline D → Q → U → P pipeline badge. Current env = purple filled, promoted envs = green filled, next promotable = orange dashed (clickable to promote), future = gray. Clicking the orange circle opens the promote dialog.

5. Config Key Promotion Done

When promoting, service config keys are copied to the target env's DynamoDB table with empty values. Admin fills in target-env-specific URLs on the Config page. Existing keys in target env are not overwritten. Lambda role has dynamodb:Query and dynamodb:PutItem permissions on the config table.

6. Replay Script Templatization In Progress

Frontend templatizes the replay script at save time: replaces config values (e.g., www.bestbuy.com) with {service.BESTBUY_USA_URL} references. At replay time in any env, templates resolve to that env's config values. Same script works everywhere — like environment variables in code.

7. Full Service Config Loading on Agent Page Done

Agent page loads all service configs (not just names) from DynamoDB at startup. Selected service's full config is passed to the agent via serviceConfig parameter. Agent uses preloaded config instead of querying DynamoDB again.

Database Evaluation
Aurora vs DynamoDB vs Hybrid — what's best for test cases at scale
P2 · 2–3 days
Current State

Test cases and run records are in Aurora Serverless v2 (PostgreSQL + pgvector). Config is in DynamoDB. Aurora has auto-pause which causes 25-30s cold starts.

Comparison
CriteriaAurora Serverless v2DynamoDBHybrid (Current + Optimized)
Cold start 25-30s on auto-pause resume 0ms always on 0-30s depends on which store
Cost (idle) $0.50/mo (auto-paused) $0/mo (on-demand mode) ~$0.25/mo
Cost (active) $15-30/mo (0.5-2 ACU) $5-15/mo (pay per request) $10-20/mo
Semantic search Native pgvector Not supported Need OpenSearch Aurora for vector search
Complex queries Full SQL JOINs, aggregates Limited pk/sk only, no JOINs SQL for reports, DynamoDB for CRUD
Multi-region Global Database (~$50/mo) Global Tables (1.5x writes) Mix of both
Report generation Excellent SQL aggregation Poor scan + filter in code Aurora for reports
Recommendation
Keep Hybrid — Optimize Aurora

Aurora is the right choice for test cases because of pgvector (semantic search), SQL aggregation (reports), and relational integrity (run records linked to test cases). Moving to DynamoDB would lose these capabilities.

Optimization: Instead of removing Aurora, fix the cold start problem:

  • Add CloudWatch synthetic canary to keep Aurora warm during business hours (8am-6pm)
  • Add retry-with-backoff in Lambda for the initial resume connection
  • Consider disabling auto-pause if usage justifies it (~$30/mo to keep warm 24/7)

DynamoDB stays for config (services, accounts) — it's already working perfectly with browser-direct access.

Failure Analysis & Logs
Screenshots, DOM state, error logs, and LLM-powered root cause analysis
P0 · 5–7 days
The Problem

When a test fails today, the user sees "FAIL" and a brief summary. No screenshots, no DOM state, no error logs. Debugging requires re-running the test with the live browser view and watching it manually. This can take 30-60 minutes per failure.

What to Capture on Failure
Screenshot at Failure Point

Playwright captures a full-page screenshot at the exact moment of failure. Stored in S3 with test run ID as key. Displayed inline in the Inventory page.

DOM Snapshot

Capture the page's HTML at failure time. Allows inspection of element visibility, missing elements, unexpected content without re-running.

Console Logs & Network Errors

Playwright can capture browser console output and failed network requests. Store as JSON — shows 404s, 500s, JS errors that caused the failure.

LLM-Powered Root Cause Analysis

When a test fails, automatically send the following to Claude Sonnet:

  • The test step that failed (action + expected result)
  • The actual error message from Playwright
  • The screenshot (as base64 image — Claude is multimodal)
  • Console errors and failed network requests
  • The previous 2-3 steps that succeeded (for context)

Claude analyzes and returns: (a) probable root cause, (b) whether it looks like a test issue vs app bug, (c) suggested fix for the test if applicable.

Implementation
1. Playwright MCP: capture artifacts on failure

Modify the Playwright MCP server to capture screenshot (page.screenshot()), DOM (page.content()), and console logs (page.on('console')) when any step fails. Upload to S3 under s3://prompt2test-artifacts/{run_id}/.

2. Agent: call Claude for root cause analysis

After failure artifacts are captured, the agent sends them to Claude with a structured prompt: "Analyze this test failure. Here are the test steps, the error, and a screenshot of the browser at failure time." Store the analysis in the run_records table.

3. UI: failure detail panel

In Inventory, clicking a failed run shows: screenshot, error message, LLM analysis, console logs, and a "Re-automate" button that takes the user back to Agent page with the test case pre-loaded for AI-assisted fix.

Report Generation
PDF/HTML reports per run, per suite, with trends and metrics
P1 · 4–5 days
Report Types
Single Run Report

Generated after each test execution. Contains: test case details, each step with PASS/FAIL, screenshots (if failed), execution time per step, overall result, and LLM analysis if applicable.

Suite Run Report

Generated after a tagged suite runs (e.g., "regression"). Contains: all test results, pass/fail counts, failed test summaries, total execution time, and comparison with previous suite run.

Trend Report

Weekly/monthly aggregate. Pass rate over time, flaky test identification (tests that flip between pass/fail), most-failed services, average execution time trends.

Implementation
1. Report Lambda

New Lambda function: report-generator. Takes a run ID or suite run ID, queries Aurora for all results, generates an HTML report, converts to PDF using Puppeteer/Playwright, uploads to S3.

2. Aurora SQL aggregation

This is why Aurora matters — trend queries like SELECT service, COUNT(*) FILTER (WHERE result='PASS') / COUNT(*) AS pass_rate FROM run_records GROUP BY service ORDER BY pass_rate are trivial in SQL but painful in DynamoDB.

3. UI: Reports tab

New tab in the sidebar. Lists past reports, filterable by date, suite, service. Download as PDF. View inline as HTML. Share via pre-signed S3 URL.

How It Works
1. Add tags array to test_cases table

PostgreSQL array column: tags TEXT[]. Supports multiple tags per test case. Indexed with GIN for fast filtering: WHERE tags @> ARRAY['regression'].

2. UI: Tag chips on test cases

In Inventory, each test case shows its tags as colored chips. Click to filter. Bulk-add/remove tags via multi-select. In Agent page, add tags when saving a new test case.

3. "Run Suite" button

New action in Inventory: "Run Suite" — select a tag (e.g., "regression"), it queues all matching test cases for execution. Creates a suite_run record that groups individual runs.

4. Suite Run record

New table: suite_runs with fields: suite_id, tag, env, team, total_tests, passed, failed, started_at, completed_at, triggered_by. Each individual run links back via suite_run_id.

Headless Suite Execution + Smart Retry
Run suites headless for speed, LLM retry on failure, email on final failure
P0 · 7–10 days
The Big Picture

This is what makes Prompt2Test a CI/CD-grade automation platform. When running a suite of deterministic replays, there's no need for a live browser — run headless for speed. When a test fails, don't just report it — let the LLM analyze and retry with an adjusted approach. Only escalate to humans when the smart retry also fails.

Headless Mode
Why Headless?

Deterministic replays don't need human observation. Running headless (no Xvfb, no noVNC, no VNC port) is 2-3x faster and uses 50% less memory. Also enables parallel execution across multiple ECS tasks.

Implementation

Add headless: true parameter to Playwright MCP. When headless, skip Xvfb/noVNC startup, launch Chromium with --headless=new. ECS task definition: reduce to 1 vCPU / 2GB (from 2 vCPU / 4GB). ~50% cost reduction per task.

Smart Retry with LLM
Step 1: Capture failure context

Same as Feature 3 — screenshot, DOM, console logs, network errors, failed step details.

Step 2: LLM analysis

Send failure context to Claude: "This deterministic replay failed at step 4. Here's the error and screenshot. Is this a flaky test (timing, animation, slow load) or a real app bug? If flaky, suggest a modified approach for retry."

Step 3: Smart retry

If Claude identifies it as flaky (e.g., "element not visible yet — likely a timing issue"), it generates a modified replay with added waits or alternative selectors. The system retries once with this adjusted approach.

Key constraint: Smart retry only modifies how to interact (waits, selectors), never what to test. The test steps remain the same — only the execution strategy adapts.

Step 4: Final failure → Email notification

If the smart retry also fails, it's likely a real bug. Send email via SES to the team/user who owns the test case. Include: test name, step that failed, screenshot, LLM analysis, and a direct link to the run record in the UI.

Parallel Execution
Run multiple tests simultaneously

For a suite of 20 tests, spin up 5 ECS tasks in parallel (configurable concurrency). Each task runs 4 tests sequentially. Total time: ~4x faster than sequential. Use an SQS queue to distribute test cases to workers.

noVNC via ALB (VPN-Friendly Live Browser View)
Route noVNC through ALB on port 443 so corporate VPNs don't block it
P1 · 2.5 days
The Problem

noVNC currently connects directly to the ECS Fargate task's public IP on port 6080 via unencrypted WebSocket (ws://). Corporate VPNs block this because: (a) non-standard port, (b) raw IP not a domain, (c) unencrypted WebSocket, (d) unknown destination.

Solution: ALB + HTTPS + Domain
Browserwss:// port 443
→
Route 53novnc.prompt2test.com
→
ALBHTTPS + ACM cert
→
ECS Taskprivate IP :6080
→
noVNC Serverlive browser stream
Why VPN Allows This
VPN CheckCurrent (Blocked)After ALB (Allowed)
Port6080 (non-standard)443 (standard HTTPS)
DestinationRaw IP (54.23.45.67)Domain (novnc.prompt2test.com)
Encryptionws:// (unencrypted)wss:// (TLS encrypted)
ProtocolUnknown WebSocketHTTPS WebSocket upgrade
Implementation Steps
1. CDK: ALB + ACM Certificate + Route 53

Create an Application Load Balancer in public subnets with HTTPS listener on port 443. ACM certificate for novnc.prompt2test.com with DNS validation. Route 53 A record pointing to the ALB. IP-type target group on port 6080.

2. Agent: Register/Deregister ECS Task as ALB Target

When start_session creates an ECS task, register the task's private IP as an ALB target (elbv2.register_targets()). When the test ends or is stopped, deregister the target. Agent returns wss://novnc.prompt2test.com instead of raw IP.

3. Security Groups

ECS tasks: allow port 6080 inbound only from ALB security group (not from anywhere). ALB: allow port 443 inbound from anywhere. No more public IP exposure on ECS tasks.

4. Frontend: Update noVNC URL

Frontend currently opens ws://IP:6080. Change to wss://novnc.prompt2test.com. The noVNC client library supports WSS natively.
