An Architecture Decision Record (ADR) captures the reasoning behind each significant technical choice -- not just what was decided, but why, what alternatives were considered, and what trade-offs were accepted. This prevents decisions from being re-litigated without understanding the original context, and helps new team members understand the platform quickly.

Each record follows the same structure: Context (the problem) → Decision (what was chosen) → Alternatives (what was considered) → Why (the reasoning) → Trade-offs (what was accepted).

ADR-01: Aurora Serverless v2 + pgvector for data storage and semantic search

Covers: test case storage / run records / vector embeddings / similarity search | Status: (/) Active

Context

The platform needs to store structured test case data (description, steps, tags, run history) and support semantic search -- finding similar test cases using natural language queries like "login tests" or "payment flow". These are two very different storage patterns: relational for CRUD, vector for similarity search. Choosing two separate systems adds operational complexity.

Decision

Use Aurora Serverless v2 PostgreSQL with the pgvector extension as the single database for both structured data and vector embeddings. Test cases are stored as rows; their embeddings (generated by Titan Embed v2) are stored as a vector column on the same row.

Alternatives Considered

OptionProsCons
Aurora v2 + pgvector (CHOSEN)One database for both SQL and vectors. No extra service. pgvector supports cosine similarity natively. Aurora Serverless scales to zero.pgvector is less scalable than dedicated vector DBs at very high volume (millions of vectors).
DynamoDB + Pinecone (REJECTED)DynamoDB is highly scalable; Pinecone is purpose-built for vectors.Two services to manage. Extra cost. No joins. Pinecone adds another external dependency and monthly cost even at low usage.
DynamoDB + OpenSearch (REJECTED)Both managed AWS services. OpenSearch supports k-NN vector search.OpenSearch minimum cost is ~$50-100/month even idle. DynamoDB has no native vector support. Complex sync between two stores.
RDS PostgreSQL (provisioned) + pgvector (REJECTED)Same pgvector capability. Predictable performance.Always-on cost even when not in use (~$30+/month minimum). No auto-pause for a dev/test platform.

Why this choice

  • Single system -- test case metadata and its vector embedding live on the same row. No sync, no dual writes, no eventual consistency issues.
  • Scales to zero -- Aurora Serverless v2 auto-pauses after 5 minutes of inactivity. For a platform that runs tests periodically (not continuously), this makes idle cost ~$0.
  • pgvector is sufficient -- at the scale of test cases (hundreds to low thousands), pgvector with an HNSW index performs sub-second similarity search. A dedicated vector DB would be over-engineering.
  • SQL for everything else -- run records, filtering by service/env/tag, aggregations -- all naturally fit SQL. DynamoDB would require duplicating data or using scan operations.

Trade-offs Accepted

  • Cold start latency -- Aurora takes 8-30 seconds to resume from paused state. First query after idle period is slow. Mitigated with retry logic in Lambda (8 retries x 8s delay).
  • pgvector scale ceiling -- if the platform ever holds millions of test vectors, pgvector will be slower than Pinecone/OpenSearch. Acceptable for current scope.
  • No offline/edge access -- Aurora is VPC-bound (accessed via Data API). No local dev database without mocking.

ADR-02: RDS Data API -- Lambda runs outside the VPC

Covers: Lambda-to-Aurora connectivity / VPC design / cold start performance | Status: (/) Active

Context

Lambda functions need to query Aurora PostgreSQL. The traditional approach is to place Lambda inside the same VPC as Aurora and use a direct TCP connection. However, this has well-known downsides. Aurora Serverless v2 supports an alternative: the RDS Data API, which exposes Aurora over HTTPS without requiring VPC membership.

Decision

Lambda functions (p2t-testcase-writer and p2t-testcase-reader) run outside any VPC and connect to Aurora exclusively through the RDS Data API (rds-data.execute_statement). No VPC configuration on the Lambda functions.

Alternatives Considered

OptionProsCons
RDS Data API (Lambda outside VPC) (CHOSEN)No VPC config on Lambda. Faster cold starts. Simpler IAM-based auth. No connection pool management.Data API adds ~10-20ms latency per call. Not supported on all Aurora configurations.
Lambda inside VPC + direct TCP (REJECTED)Lower latency per query. Standard PostgreSQL driver (psycopg2). Connection pooling possible.Lambda cold starts add 1-3 seconds for VPC ENI attachment. Requires NAT Gateway for internet access (~$32/month). Complex subnet/SG config.
Lambda inside VPC + RDS Proxy (REJECTED)Handles connection pooling. Reduces Aurora connection exhaustion.RDS Proxy costs ~$20+/month. Adds another component. Still requires Lambda in VPC with all its cold start issues.

Why this choice

  • No NAT Gateway cost -- Lambda inside a VPC needs a NAT Gateway to reach Bedrock, Secrets Manager, and other AWS services. NAT Gateway costs ~$32/month minimum, plus data transfer.
  • Faster cold starts -- VPC-attached Lambda cold starts are 1-3 seconds longer due to ENI creation. Outside VPC, cold starts are under 500ms.
  • Simpler architecture -- no subnet routing, no security group rules between Lambda and Aurora, no connection pool management.
  • Aurora Serverless v2 is designed for this -- the Data API is the recommended access pattern for serverless Aurora.

Trade-offs Accepted

  • ~10-20ms extra latency per query vs direct TCP. Acceptable for a test management platform.
  • No persistent connections -- each Data API call is stateless. Can't use transactions spanning multiple Lambda invocations.
  • Data API payload limit -- 1MB response limit per call. Fine for test case data but would need pagination for very large result sets.

ADR-03: Aurora auto-pause enabled -- optimise for cost, not always-on availability

Covers: Aurora Serverless v2 scaling / idle cost / cold start UX impact | Status: (!) Trade-off

Context

Aurora Serverless v2 supports auto-pause: if the cluster receives no queries for a configurable period (set to 5 minutes here), it scales down to 0 ACUs and stops billing for compute. The next query wakes it up.

Decision

Aurora auto-pause is enabled with a 5-minute idle threshold. All Lambda functions that query Aurora implement a retry loop (8 retries, 8-second delay) that catches DatabaseResumingException and waits for the cluster to resume.

Why this choice

  • Cost -- Aurora Serverless v2 at minimum capacity (0.5 ACU) costs ~$0.12/hour = ~$87/month even if unused. Auto-pause brings idle cost to $0.
  • Transparent to users -- the retry logic in Lambda absorbs the resume delay. The UI shows a loading spinner, not an error.
  • Appropriate for usage pattern -- test case management is not a 24/7 high-traffic system. Testers use it in bursts during working hours.

Trade-offs Accepted

  • First-request latency after idle -- 8-30 seconds for Aurora to resume.
  • Retry complexity in Lambda -- every Lambda that queries Aurora must implement the retry pattern.
  • Not suitable for production SLA -- if this platform is ever used in a production context with SLA requirements, auto-pause should be disabled.

ADR-04: Bedrock AgentCore for the AI agent runtime

Covers: agent hosting / tool use / long-running conversations / session management | Status: (/) Active

Context

The AI agent needs to run a Python container (built with the Strands SDK) that: maintains multi-turn conversation context, calls Bedrock Claude for reasoning, uses tools (MCP, SSM lookups, Lambda invocations), and streams responses back to the browser.

Decision

Use Bedrock AgentCore to host the agent container. AgentCore manages the container lifecycle, session state, and streaming.

Alternatives Considered

OptionProsCons
Bedrock AgentCore (CHOSEN)Built for agentic workloads. Handles session state, streaming, container lifecycle. Integrates natively with Bedrock models and MCP protocol.New service, limited CloudFormation support. Less community documentation.
Lambda (Python) (REJECTED)Simple. Cheap. Auto-scaling. CDK native.15-minute timeout. Stateless. No native streaming. Cold starts with large dependencies.
ECS Fargate (always-on) (REJECTED)Full container control. No timeout limits.Always-on cost (~$30-50/month). Needs load balancer. Not purpose-built for AI agents.
Bedrock Agents (managed) (REJECTED)Fully managed. No container to maintain.Limited to Bedrock-defined action groups. Can't run arbitrary Python code or custom tool integrations.

Why this choice

  • Purpose-built for agentic workloads -- AgentCore handles session persistence, response streaming, container warm-up, and MCP protocol integration.
  • No always-on cost -- unlike a standard ECS service, AgentCore only runs when invoked.
  • Strands SDK alignment -- the agent is built with AWS Strands, which is designed to deploy to AgentCore.
  • Streaming to browser -- AgentCore's streaming API lets the UI display the agent's response token by token.

Trade-offs Accepted

  • No CloudFormation support -- AgentCore must be created via CLI (one manual step).
  • New service risk -- AgentCore is relatively new. APIs may change.
  • ECR image dependency -- AgentCore must be created after the CodePipeline pushes the first image to ECR.

ADR-05: ECS Fargate on-demand for Playwright browser automation

Covers: test execution runtime / browser container / noVNC live view / cost model | Status: (/) Active

Context

Running Playwright requires a full browser (Chromium) -- a heavy workload (200MB+ memory). It also needs to serve a live noVNC view. Lambda cannot run a browser, and an always-on container would waste money between test runs.

Decision

Playwright runs in an ECS Fargate task launched on-demand per test execution (ARM64, 2 vCPU, 4GB RAM). The task starts when a user clicks Automate or Run Test, serves a live browser view via noVNC over WebSocket, executes the test, records the result, then terminates.

Alternatives Considered

OptionProsCons
ECS Fargate on-demand task (CHOSEN)Full container. Runs real browser. noVNC live view. Scales to zero. ~$0.10 per test run.30-60 second cold start. More complex than Lambda.
Lambda (headless Chrome via layer) (REJECTED)Faster start. Cheaper per invocation.Lambda has 10GB storage and 3GB RAM limit. 15-minute timeout. No live view possible.
ECS Fargate always-on service (REJECTED)No cold start. Always ready.Costs ~$30-50/month even when no tests are running.
EC2 instance (dedicated) (REJECTED)Full control. Persistent storage.Always-on cost. Manual maintenance. No auto-scaling.

Why this choice

  • Full browser environment -- Playwright needs a real Chromium. Only a full container provides this reliably.
  • Live noVNC view -- requires a persistent WebSocket connection from a running container.
  • Pay per run -- a 2-minute test run costs approximately $0.04. Zero cost between runs.
  • Isolation -- each test run gets a fresh container, fresh browser state.

Trade-offs Accepted

  • 30-60 second startup time -- mitigated by showing a progress indicator in the UI.
  • Image pull latency -- large Docker image (2-3GB). Fargate caches images.
  • No parallel run limit defined -- currently no cap on simultaneous Fargate tasks.

ADR-06: ARM64 (Graviton) for ECS task definition

Covers: compute architecture / cost / Docker build compatibility | Status: (/) Active

Decision

ECS task definition uses ARM64 (Graviton2). Fargate Graviton2 pricing is ~20% cheaper than x86_64.

Why this choice

  • ~20% cost reduction per vCPU-hour and per GB-hour.
  • Playwright supports ARM64 -- stable ARM64 Linux builds. No compatibility issues.
  • Performance parity -- for I/O-bound workloads like browser automation.

Trade-offs Accepted

  • Build pipeline must target ARM64 -- CodeBuild must use LINUX_ARM_CONTAINER, or use Docker buildx.
  • Some native packages may lack ARM64 builds -- would need to revisit if future dependencies don't support ARM64.

ADR-07: 4 separate GitHub repos -- not a monorepo

Covers: repo structure / deploy cadence / pipeline triggers / team ownership | Status: (/) Active

Decision

Four separate GitHub repositories: Prompt2TestUI (React + CDK), Prompt2TestLambda (Python), Prompt2TestAgent (agent container), Prompt2TestPlaywrightMCP (Playwright container). Each repo has its own pipeline.

Why this choice

  • Independent deploy cadences -- the UI changes frequently; Lambda logic changes occasionally; the Playwright container rarely changes.
  • Clear boundaries -- each repo corresponds to one deploy target.
  • Smaller blast radius -- a broken push to one repo doesn't affect others.
  • Different tech stacks -- TypeScript/React, Python, and Dockerfile-based containers have no shared build tooling.

Trade-offs Accepted

  • Cross-repo changes need coordination -- if a new Lambda API requires a UI change, two PRs in two repos must be coordinated.
  • 4 repos to keep up to date -- each repo needs its own pipeline setup and README.

ADR-09: Amazon Cognito for authentication

Covers: user login / session tokens / browser-to-AWS SDK calls / IAM federation | Status: (/) Active

Decision

Use Cognito User Pool for login and Cognito Identity Pool to exchange the JWT for temporary AWS credentials. The browser SDK uses these credentials to call Lambda, AgentCore, and ECS directly -- no backend proxy needed.

Why this choice

  • Browser-to-AWS without a backend proxy -- Cognito Identity Pool issues temporary STS credentials to the browser.
  • No credentials in frontend code -- credentials are short-lived (1 hour) and scoped.
  • Fully managed -- Cognito handles password hashing, session tokens, MFA, and JWT validation.
  • CDK native -- User Pool, Identity Pool, and Auth Role are all provisionable with CDK.

ADR-10: Lambda deployed via separate CodePipeline, not bundled in CDK

Covers: Lambda code lifecycle / CDK asset management / repo separation | Status: (/) Active

Decision

CDK creates Lambda functions with placeholder code (a stub that returns a 503). A separate CodePipeline (prompt2test-lambda) watches Prompt2TestLambda on GitHub and runs aws lambda update-function-code on every push.

Why this choice

  • Lambda code has its own cadence -- developers should not need to run cdk deploy for a Lambda bug fix.
  • CDK deploy is slow -- cdk deploy takes 5-10 minutes; update-function-code takes under 30 seconds.
  • Separation of concerns -- infrastructure team manages CDK; backend team manages Lambda code.

Covers: API design / browser-to-Lambda call pattern / latency / cost | Status: (/) Active

Decision

The React UI calls Lambda using lambda.invoke() from the AWS SDK with Cognito-issued credentials. No API Gateway. No HTTP REST or GraphQL endpoint.

ADR-12: SSM Parameter Store for infrastructure configuration only

Covers: ECS cluster config / Playwright subnet/SG / Aurora connection details / infra-only params | Status: (/) Active

Decision

SSM stores only 4 Playwright/ECS infra params written by CDK at deploy time: /prompt2test/playwright/cluster-name, task-definition-family, subnet-ids, and security-group-id. Also /prompt2test/aurora/* used by Lambda. No user-managed config in SSM. Service configs moved to DynamoDB (ADR-13). Team membership moved to Cognito Groups (ADR-14).

ADR-13: DynamoDB for service and account configuration

Covers: service URLs / API keys / test accounts / per-team per-env config / Config and Accounts UI | Status: (/) Active

Decision

DynamoDB table prompt2test-config with composite keys. Services: pk=SERVICE#{team}#{env}, sk={svc}#{key}, val={value}. Accounts: pk=ACCOUNT#{env}, sk={id}#{NAME|CODE}, val={value}. The browser calls DynamoDB directly via Cognito IAM credentials.

Why this choice

  • Clean team isolation -- pk=SERVICE#{team}#{env} means each team's config is in its own partition.
  • Direct browser access -- no Lambda needed for config reads/writes.
  • Schema-less -- teams can add arbitrary service parameters without schema migrations.
  • Free tier -- 25 GB storage, 25 RCU/WCU included free.
  • No path management -- composite key is explicit, no double-slash edge cases.

ADR-14: Cognito Groups for team membership

Covers: team definitions / member assignments / team resolution at login / Members page admin UI | Status: (/) Active

Decision

Each team = a Cognito Group in the User Pool. Team resolved from cognito:groups JWT claim immediately after login. Admin creates groups and adds users via the Members page using Cognito IDP APIs.

Why this choice

  • Cognito already owns identity -- adding groups is the natural extension.
  • JWT claim = zero latency -- team is available immediately after login.
  • Standard Cognito IDP APIs -- ListGroups, CreateGroup, AdminAddUserToGroup.
  • Admin UX -- Members page provides full team/member management UI.

Trade-offs Accepted

  • Flat groups only -- no nesting.
  • Multi-group users -- app uses the first group from the JWT claim.
  • Case-sensitive group names -- TeamA and teama are different groups.
