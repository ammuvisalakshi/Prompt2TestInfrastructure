Feature 1: Environment Promotion (SHIPPED)

Promote test cases from DEV to higher environments as deterministic replay only.

The Problem

Today, all environments use the same AI-powered authoring flow. In higher environments (QA, UAT, PROD), we need deterministic, repeatable execution -- not AI re-interpretation. A test that passed in DEV should execute the exact same steps in QA.

Promotion Flow

DEV (AI authors + tests)→Review and Approve (QA lead signs off)→QA (deterministic replay)→Validate and Approve→UAT (deterministic replay)→PROD (deterministic replay)

Design Decisions

  • DEV = Full AI Mode: AI can author, modify, and iterate on test plans. This is the experimentation environment.
  • QA / UAT / PROD = Replay Only: Promoted test cases are locked. No AI re-generation. Only deterministic replay using the exact saved steps. UI hides the "Author Agent" chat in these environments.

Implementation (SHIPPED)

1. {service.KEY} Config References in Plan Steps (/) Done

Agent generates plan steps with {service.BESTBUY_USA_URL} template references instead of hardcoded URLs. The LLM's response is post-processed: config values are replaced with {service.KEY} templates automatically. UI shows resolved values with the template in small gray text below.

2. promote_test_case Lambda Action (/) Done

New action in p2t-testcase-writer Lambda. Copies test case to target env (ID: TC-xxx-qa), strips action_resolved/expected_resolved fields (templates only), promotes service config keys to target env DynamoDB with empty values, and updates source test case's promoted_to array.

3. Sequential Promotion (DEV → QA → UAT → PROD) (/) Done

Promotion is sequential only: DEV can promote to QA, QA to UAT, UAT to PROD. No skipping environments. PROD has no promote option.

4. Promotion Pipeline Badge on Inventory UI (/) Done

Each test case row shows an inline D → Q → U → P pipeline badge. Current env = purple filled, promoted envs = green filled, next promotable = orange dashed (clickable to promote), future = gray.

5. Config Key Promotion (/) Done

When promoting, service config keys are copied to the target env's DynamoDB table with empty values. Admin fills in target-env-specific URLs on the Config page. Existing keys in target env are not overwritten.

6. Replay Script Templatization (!) In Progress

Frontend templatizes the replay script at save time: replaces config values with {service.KEY} references. At replay time in any env, templates resolve to that env's config values.

7. Full Service Config Loading on Agent Page (/) Done

Agent page loads all service configs (not just names) from DynamoDB at startup. Selected service's full config is passed to the agent via serviceConfig parameter.

Aurora vs DynamoDB vs Hybrid -- what's best for test cases at scale. Priority: P2 | Effort: 2-3 days

Comparison

Recommendation: Keep Hybrid -- Optimize Aurora

Aurora is the right choice for test cases because of pgvector (semantic search), SQL aggregation (reports), and relational integrity. Optimization: fix the cold start problem with CloudWatch canary to keep Aurora warm during business hours, add retry-with-backoff in Lambda, or consider disabling auto-pause (~$30/mo to keep warm 24/7). DynamoDB stays for config (services, accounts).

Feature 3: Failure Analysis and Logs

Screenshots, DOM state, error logs, and LLM-powered root cause analysis. Priority: P0 | Effort: 5-7 days

What to Capture on Failure

  • Screenshot at Failure Point: Playwright captures full-page screenshot. Stored in S3 with test run ID as key.
  • DOM Snapshot: Capture page HTML at failure time. Allows inspection without re-running.
  • Console Logs and Network Errors: Capture browser console output and failed network requests. Store as JSON.

LLM-Powered Root Cause Analysis

When a test fails, send to Claude Sonnet: the failed step, the error message, the screenshot (base64 image), console errors, and the previous 2-3 steps. Claude returns: (a) probable root cause, (b) test issue vs app bug classification, (c) suggested fix if applicable.

Feature 4: Report Generation

PDF/HTML reports per run, per suite, with trends and metrics. Priority: P1 | Effort: 4-5 days

Report Types

  • Single Run Report: Test case details, each step with PASS/FAIL, screenshots if failed, execution time, LLM analysis.
  • Suite Run Report: All test results, pass/fail counts, failed test summaries, comparison with previous suite run.
  • Trend Report: Pass rate over time, flaky test identification, most-failed services, execution time trends.

Implementation

  • Report Lambda: Takes a run ID or suite run ID, queries Aurora, generates HTML report, converts to PDF, uploads to S3.
  • Aurora SQL aggregation: Trend queries are trivial in SQL.
  • UI Reports tab: New tab in sidebar. Lists past reports, filterable by date/suite/service. Download as PDF or view inline.

Feature 6: Headless Suite Execution + Smart Retry

Run suites headless for speed, LLM retry on failure, email on final failure. Priority: P0 | Effort: 7-10 days

Execution Flow

Trigger Suite (tag: "regression")→Queue Tests (12 test cases)→Headless Replay (no noVNC)→10 PASS→2 FAIL→LLM Analyzes→Smart Retry→Still fails? Send email

Headless Mode

Deterministic replays don't need human observation. Running headless (no Xvfb, no noVNC) is 2-3x faster and uses 50% less memory. ECS task: reduce to 1 vCPU / 2GB. ~50% cost reduction per task.

Smart Retry with LLM

  1. Capture failure context: Screenshot, DOM, console logs, network errors.
  2. LLM analysis: Send to Claude -- "Is this a flaky test (timing, animation, slow load) or a real app bug?"
  3. Smart retry: If Claude identifies flaky, it generates a modified replay with added waits or alternative selectors. Key constraint: Only modifies how to interact, never what to test.
  4. Final failure → email: If smart retry also fails, send email via SES with test name, step, screenshot, analysis, and link.

Parallel Execution

For a suite of 20 tests, spin up 5 ECS tasks in parallel (configurable). Each runs 4 tests sequentially. ~4x faster. Use SQS queue to distribute.

Architecture: Suite Execution Engine

UI: Run Suite→Lambda: Orchestrator→SQS Queue→N x ECS Tasks (headless)→Results → Aurora→Failures → LLM→SES Email

Feature 7: noVNC via ALB (VPN-Friendly Live Browser View)

Route noVNC through ALB on port 443 so corporate VPNs don't block it. Priority: P1 | Effort: 2.5 days

The Problem

noVNC currently connects directly to ECS Fargate task's public IP on port 6080 via unencrypted WebSocket (ws://). Corporate VPNs block this because: (a) non-standard port, (b) raw IP not a domain, (c) unencrypted WebSocket, (d) unknown destination.

Solution: ALB + HTTPS + Domain

Browser (wss:// port 443)→Route 53 (novnc.prompt2test.com)→ALB (HTTPS + ACM cert)→ECS Task (private IP :6080)→noVNC Server

Why VPN Allows This

VPN CheckCurrent (Blocked)After ALB (Allowed)
Port(x) 6080 (non-standard)(/) 443 (standard HTTPS)
Destination(x) Raw IP (54.23.45.67)(/) Domain (novnc.prompt2test.com)
Encryption(x) ws:// (unencrypted)(/) wss:// (TLS encrypted)
Protocol(x) Unknown WebSocket(/) HTTPS WebSocket upgrade

Implementation

  1. CDK: ALB + ACM Certificate + Route 53 -- ALB in public subnets with HTTPS listener on 443. ACM certificate for novnc.prompt2test.com. Route 53 A record. IP-type target group on port 6080.
  2. Agent: Register/Deregister ECS Task as ALB Target -- When start_session creates a task, register private IP as ALB target. On test end, deregister.
  3. Security Groups -- ECS tasks: allow port 6080 only from ALB SG. ALB: allow port 443 from anywhere.
  4. Frontend: Update noVNC URL -- Change from ws://IP:6080 to wss://novnc.prompt2test.com.
