Prompt2Test -- New AWS Account Deployment Guide

Complete step-by-step walkthrough with expected output after every step. Brand new AWS account → fully running platform.

Estimated Time: 45-60 minutes | AWS Region: us-east-1 | Updated: 2026-03-25

infrastructure/ folder = CDK code (the automated installer) -- TypeScript code that, when you run one command (cdk deploy), automatically creates all 16 AWS resources (Cognito, Aurora, Lambda, ECS, ECR, SSM, CodePipeline, Amplify, IAM roles) in your account.

Manual steps (3 total) -- Three things CDK cannot automate: enabling Bedrock model access, creating the AgentCore runtime (no CloudFormation support yet), and connecting Amplify to GitHub (requires OAuth).

AWS Account Requirements

RequirementWhy
New AWS account with a credit card on fileRequired to provision any paid services
AdministratorAccess IAM policy attached to your userCDK creates IAM roles, VPC, 15+ services

If you are the root user of a brand new account, you already have full access. If you created a separate IAM user, attach AdministratorAccess first: AWS Console → IAM → Users → your user → Add permissions → AdministratorAccess.

Phase 1: Local Machine Setup

Step 1: Install AWS CLI

Download from aws.amazon.com/cli. After install run: aws --version

(/) You should see version 2.x.x

Step 2: Install Node.js (version 18 or higher)

Download LTS from nodejs.org. Run: node --version

(/) Any version v18.x.x or higher is fine.

Step 3: Install AWS CDK globally

npm install -g aws-cdk then cdk --version

(/) Any 2.x.x version is good.

Step 4: Attach AdministratorAccess and get your access key

Part A -- Attach AdministratorAccess:

  1. Log into the new AWS account at console.aws.amazon.com
  2. Search for IAM → click Users → click your username
  3. Click Add permissions → Attach policies directly
  4. Search for AdministratorAccess → check the box → Next → Add permissions

Part B -- Create an access key:

  1. Click your account name (top-right) → Security credentials
  2. Scroll to Access keys → Create access key
  3. Select Command Line Interface (CLI) → confirm → Next
  4. Create access key -- copy both Access key ID and Secret access key

Step 5: Configure AWS CLI

Run aws configure and enter: Access Key ID, Secret Access Key, Default region: us-east-1, Output format: json

Verify: aws sts get-caller-identity

(/) The Account number must be your new AWS account ID.

Step 6: Clone the Prompt2TestUI repository

git clone https://github.com/YOUR_GITHUB_USERNAME/Prompt2TestUI then cd Prompt2TestUI

You only need Prompt2TestUI locally because it contains the CDK infrastructure code (infrastructure/ folder). The other 3 repos are pulled directly from GitHub by the CodePipelines.

Phase 2: Prepare the AWS Account -- Manual Steps in Console

Step 7: Enable Bedrock Model Access

Start this step first. Bedrock model access approval usually takes 1-5 minutes.
  1. AWS Console → search for Bedrock → click Amazon Bedrock
  2. Left sidebar → Model access
  3. Click Manage model access
  4. Check: Claude Sonnet 4.5 (Anthropic, cross-region inference profile) and Titan Embed Text v2 (Amazon)
  5. Click Save changes

(/) Both should show "Access granted" after 1-5 minutes.

Step 8: Bootstrap CDK in the new account

Get your account ID: aws sts get-caller-identity --query Account --output text

Run: cdk bootstrap aws://123456789012/us-east-1 (replace with your account ID)

(/) Terminal shows "bootstrapped".

Step 9: Create a GitHub CodeStar Connection

  1. AWS Console → Developer Tools → Settings → Connections
  2. Click Create connection
  3. Select GitHub → name it Prompt2TestGitHub → Connect to GitHub
  4. GitHub popup -- click Authorize AWS Connector for GitHub
  5. Back in AWS, click Connect
  6. Click on your new connection → copy the Connection ARN

(/) Status must be Available. Save the Connection ARN.

Phase 3: CDK Deploy -- Creates All 17 AWS Resources Automatically

Step 10: Configure cdk.json

Open Prompt2TestUI/infrastructure/cdk.json and replace the two placeholder values:

  • githubOwner -- your GitHub username
  • githubConnectionArn -- the ARN from Step 9

Step 11: Run CDK Deploy

cd infrastructure → npm install → cdk deploy

CDK will show a summary of IAM permissions and ask for confirmation. Type y and press Enter. Takes 5-10 minutes.

(/) Terminal shows "Prompt2TestStack" with outputs including UserPoolId, UserPoolClientId, IdentityPoolId, Region, AgentCoreRoleArn, AgentEcrUri, AmplifyAppId.

Copy and save ALL of these Output values. You can always retrieve them later from AWS Console → CloudFormation → Prompt2TestStack → Outputs tab.

Phase 4: Deploy Lambda Code and Build Docker Images via CodePipeline

Step 12: Trigger all 3 CodePipelines

aws codepipeline start-pipeline-execution --name prompt2test-lambda

aws codepipeline start-pipeline-execution --name prompt2test-playwright-mcp

aws codepipeline start-pipeline-execution --name prompt2test-agent

Monitor in AWS Console → CodePipeline. Wait until all three show Succeeded (5-10 minutes each).

Phase 5: Create Bedrock AgentCore Runtime -- Manual CLI Step

Step 13: Create the AgentCore Runtime

Use the ECR URI and IAM role ARN from CDK outputs:

aws bedrock-agentcore-control create-agent-runtime --agent-runtime-name Prompt2TestAgent --agent-runtime-artifact '{"containerConfiguration":{"image":"YOUR_OutAgentEcrUri:latest"}}' --role-arn YOUR_OutAgentCoreRoleArn --network-configuration '{"networkMode":"PUBLIC"}'

(/) Returns JSON with agentRuntimeArn and status "CREATING". Poll until ACTIVE:

aws bedrock-agentcore-control get-agent-runtime --agent-runtime-id XXXXXXXXXX --query 'status' --output text

Copy the agentRuntimeArn value. You need it in Step 14.

Phase 6: Connect Amplify to GitHub -- Manual Console Step

Step 14: Connect Amplify to GitHub and trigger the first build

  1. AWS Console → AWS Amplify
  2. Click on Prompt2TestUI (created by CDK)
  3. Click Connect branch
  4. Select GitHub → authorize → select Prompt2TestUI repo → branch: master
  5. Go to Environment variables, find VITE_AGENT_RUNTIME_ARN and replace with the ARN from Step 13
  6. Click Save and deploy

(/) Build completes: Provision → Build → Deploy → Verify. App URL shown at top.

All other environment variables (Cognito IDs, Identity Pool ID, Region) were already set by CDK. You only need to set VITE_AGENT_RUNTIME_ARN.

Phase 7: Add Your Services and Create Admin User

Step 15: Create teams and configure services via the UI

Members page → Create teams and invite users:

  1. Go to the Members page (admin only)
  2. Click Create Team → enter team name (e.g. teamA)
  3. Click Invite User → enter email → user created in Cognito and added to team group

Config and Accounts page → Add service parameters:

  1. Go to Config and Accounts page
  2. Select environment (e.g. dev)
  3. Click Add Service → enter service name (e.g. MyService)
  4. Add parameters: URL, API_KEY, or any key-value pair
  5. Parameters save to DynamoDB with key pk=SERVICE#{team}#{env}, sk={svc}#{key}

Step 16: Create your first admin user and the admin Cognito Group

Create admin group: aws cognito-idp create-group --user-pool-id YOUR_OutUserPoolId --group-name admin --region us-east-1

Create admin user: aws cognito-idp admin-create-user --user-pool-id YOUR_OutUserPoolId --username admin@yourcompany.com --temporary-password TempPass123! --user-attributes Name=email,Value=admin@yourcompany.com Name=email_verified,Value=true

Add to admin group: aws cognito-idp admin-add-user-to-group --user-pool-id YOUR_OutUserPoolId --username admin@yourcompany.com --group-name admin --region us-east-1

(/) User status: FORCE_CHANGE_PASSWORD (expected -- user sets password on first login)

Phase 8: Verify the Full Platform End-to-End

Step 17: Log in and test the full flow

  1. Open your Amplify URL in a browser
  2. Log in with admin@yourcompany.com and temporary password TempPass123!
  3. Set a new password when prompted
  4. Go to Config and Accounts tab -- add a service via the UI
  5. Go to Agent tab → select environment and service → type a test prompt → agent should respond
  6. Save the test case → click View Test Case
  7. Click Automate → live browser window opens
  8. Click Save Steps to persist playwright_calls
  9. Click Run Test → deterministic replay → PASS or FAIL
  10. Go to Inventory tab → test case should appear with run result

(/) Login page loads = Amplify working | Password change = Cognito working | Services appear = DynamoDB working | Agent responds = AgentCore active | Test case saves = Lambda + Aurora working | Live browser = ECS Fargate + noVNC working | Steps captured = playwright_calls saved | PASS/FAIL recorded = Full end-to-end complete
