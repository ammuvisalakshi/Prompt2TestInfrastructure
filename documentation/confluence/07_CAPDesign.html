Prompt2Test -- CAP Design Improvements

Architectural enhancements to make the platform Highly Available, Strongly Consistent, and Partition Tolerant. Organized into 4 tiers from quick wins to full multi-region active-active.

Tags: CAP Theorem | High Availability | Multi-Region | Disaster Recovery | Production Hardening | us-east-1 + us-west-2

CAP Theorem -- Quick Primer

The CAP theorem states that a distributed system can guarantee at most two of three properties simultaneously. In practice, network partitions are inevitable, so the real choice is between CP (consistent but may reject requests) and AP (available but may serve stale data).

PropertyDescriptionExamples in Prompt2Test
C -- ConsistencyEvery read returns the most recent write. All nodes see the same data at the same time.Aurora (single writer), DynamoDB strong reads
A -- AvailabilityEvery request receives a response. System never refuses a request.DynamoDB, CloudFront, Lambda
P -- Partition ToleranceSystem continues to operate despite network partitions between nodes or regions.DynamoDB Global Tables, Aurora Global DB

Tier 1 -- Quick Wins (HA Within Region) -- 1-2 days effort

1.1 -- DynamoDB: Enable Strongly Consistent Reads

Agent reads service config before every test. Currently uses eventually consistent reads.

Implementation: Add ConsistentRead: true to all QueryCommand and GetCommand calls.

Trade-off: 2x read cost, slightly higher latency (~5ms). CAP Impact: Moves config reads from AP to CP.

1.2 -- Aurora: Verify Multi-AZ Deployment

Ensure writer + at least 1 reader instance in different AZs. Set min capacity = 0.5 ACU for reader.

Trade-off: ~$30/mo additional cost. CAP Impact: Improves Availability -- survives single AZ failure.

1.3 -- ECS: Multi-AZ Task Placement

Ensure subnets span 2+ AZs. Add placementStrategies: [{ type: 'spread', field: 'attribute:ecs.availability-zone' }].

Trade-off: None -- no additional cost. CAP Impact: Improves Availability of browser test sessions.

1.4 -- Lambda: Reserved Concurrency + Provisioned

Set reservedConcurrency: 5 for reader/writer. Consider provisionedConcurrency: 1 for the reader.

Trade-off: Provisioned: ~$15/mo. Reserved: free. CAP Impact: Improves Availability under load.

1.5 -- CloudWatch Alarms + SNS Notifications

Create alarms for: Aurora connection errors, Lambda error rate > 5%, ECS task launch failures, Bedrock throttling. Route to SNS → email/Slack.

Trade-off: ~$1/mo. CAP Impact: Enables fast detection of availability issues.

Tier 2 -- Production Hardening -- 1-2 weeks effort

2.1 -- API Gateway in Front of Lambda

Create API Gateway REST API with /test-cases, /run-records endpoints. Add usage plans, throttling (100 req/s), and Cognito authorizer.

Trade-off: ~50ms latency. ~$3.50/million requests. CAP Impact: Improves Availability via throttling and Consistency via validation.

2.2 -- Dead Letter Queues for Lambda

Add SQS DLQ to both writer and reader Lambdas. Alarm when DLQ depth > 0. Add retry Lambda.

Trade-off: ~$0.40/million SQS requests. CAP Impact: Improves Consistency -- writes never lost.

2.3 -- WAF on CloudFront

Attach AWS WAF with managed rule groups. Rate-based rule: 2000 req/5min per IP.

Trade-off: ~$5/mo. CAP Impact: Protects Availability from DDoS.

2.4 -- Secrets Manager Auto-Rotation

Enable automatic rotation every 30 days. Trade-off: ~$0.40/secret/month.

2.5 -- Aurora Auto-Pause Retry Pattern

Exponential retry with backoff in Lambda. CloudWatch synthetic canary to ping Aurora every 4 minutes during business hours.

Trade-off: Canary: ~$3/mo. CAP Impact: Dramatically improves Availability during business hours.

2.6 -- S3 Cross-Region Replication

Enable S3 CRR to us-west-2 for pipeline artifact bucket.

Trade-off: ~$0.023/GB/month for replica. CAP Impact: Enables Partition Tolerance for CI/CD artifacts.

Tier 3 -- Multi-Region (Partition Tolerance) -- 2-4 weeks effort

3.1 -- DynamoDB Global Tables

Replicate prompt2test-config to us-west-2. Both regions can read and write.

Trade-off: 1.5x write cost. ~1s replication lag. CAP Impact: Full AP, partition tolerant.

3.2 -- Aurora Global Database

Read replica cluster in us-west-2. Primary handles writes. Can promote in <1 minute during failover.

Trade-off: ~$40-60/mo for secondary cluster. CAP Impact: CP with HA, partition tolerant with manual failover.

3.3 -- ECS Cluster in us-west-2

Second ECS Fargate cluster with identical task definitions. Browser sessions can run in either region.

Trade-off: ECR replication: ~$0.10/GB. No idle cost. CAP Impact: Full AP, partition tolerant.

3.4 -- Route 53 Health Checks + DNS Failover

Failover routing: primary = us-east-1, secondary = us-west-2. TTL = 60s.

Trade-off: ~$1/health check/month. 60-90s failover. CAP Impact: Automatic Partition Tolerance at DNS layer.

3.5 -- Cognito Backup and Custom Sync

Option A: Pre-issue long-lived JWT refresh tokens (24h). Option B: Backup User Pool in us-west-2 with Lambda sync. Option C: Move to multi-region auth provider.

Trade-off: Option A is 1 day. Option B is ~1 week. Option C requires migration.

3.6 -- Bedrock Cross-Region Fallback

If Bedrock us-east-1 returns 429 or 5xx, retry against us-west-2 endpoint.

Trade-off: ~50ms cross-region latency. CAP Impact: AP, partition tolerant for AI layer.

Tier 4 -- Active-Active and Platform Maturity -- 1-2 months effort

4.1 -- AWS Global Accelerator

Anycast IPs and automatic failover at TCP layer. Failover in <30s vs 60-90s for DNS.

4.2 -- Multi-Region CI/CD

Single git push triggers parallel builds and deployments to us-east-1 and us-west-2.

4.3 -- Multi-Account Strategy

Separate AWS accounts for dev, staging, and production. Blast radius isolation.

4.4 -- Chaos Engineering

Use AWS Fault Injection Simulator (FIS) to simulate: AZ impairment, RDS failover, Lambda throttling, ECS task failures. Run monthly in staging.
