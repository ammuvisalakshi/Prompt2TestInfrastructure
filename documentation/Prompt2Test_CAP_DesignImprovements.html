Prompt2Test — CAP Design Improvements

Architectural enhancements to make the platform Highly Available, Strongly Consistent, and Partition Tolerant. Organized into 4 tiers from quick wins to full multi-region active-active.

CAP Theorem — Quick Primer

The CAP theorem states that a distributed system can guarantee at most two of three properties simultaneously. In practice, network partitions are inevitable, so the real choice is between CP (consistent but may reject requests) and AP (available but may serve stale data).

C Consistency

Every read returns the most recent write. All nodes see the same data at the same time.

Aurora (single writer) DynamoDB strong reads
A Availability

Every request receives a response (success or failure). System never refuses a request.

DynamoDB CloudFront Lambda
P Partition Tolerance

System continues to operate despite network partitions between nodes or regions.

DynamoDB Global Tables Aurora Global DB

Critical Finding

The platform is single-region (us-east-1). If that region experiences an outage, the entire system is unavailable — no login, no test creation, no execution, no data access. CloudFront is the only partition-tolerant component.

Recommended CAP Strategy for Prompt2Test

Since network partitions are inevitable in distributed systems, we must choose between CP and AP per component based on business requirements:

Data / OperationStrategyRationale
Service Config reads (DynamoDB) CP — Strong Consistency Agent MUST see latest URLs and credentials. Stale config = test runs against wrong environment.
Test case writes (Aurora) CP — Strong Consistency Cannot lose test results. A PASS/FAIL must be durably written before confirming to user.
Test case reads / listing (Aurora) AP — Eventual Consistency OK Inventory listing can lag by seconds. User won't notice a 1-2s delay in seeing new test cases.
Browser sessions (ECS Fargate) AP — Availability First Better to start a test in a degraded state than refuse the request entirely.
Authentication (Cognito) CP — Strong Consistency Auth tokens must be valid. Cannot allow stale or forged credentials.
AI Agent responses (Bedrock) AP — Availability First If primary model is unavailable, fall back to a simpler model rather than failing.
Run records (Aurora) CP — Strong Consistency Execution results are the source of truth. Must not be lost or duplicated.
1 Tier 1 — Quick Wins (HA Within Region) 1–2 days effort

1.1 — DynamoDB: Enable Strongly Consistent Reads

Agent reads service config before every test. Currently uses eventually consistent reads, which may return stale URLs or credentials if config was just updated.

Implementation: Add ConsistentRead: true to all QueryCommand and GetCommand calls in agent's get_service_config tool and Lambda functions.
Trade-off: 2x read cost, slightly higher latency (~5ms). Worth it for correctness.
CAP Impact: Moves config reads from AP to CP.

1.2 — Aurora: Verify Multi-AZ Deployment

Aurora Serverless v2 should have instances in at least 2 AZs. If only 1 instance exists, an AZ failure takes down the database.

Implementation: In CDK, ensure writer + at least 1 reader instance in different AZs. Set min capacity = 0.5 ACU for the reader to minimize cost.
Trade-off: ~$30/mo additional cost for the reader instance.
CAP Impact: Improves Availability — survives single AZ failure.

1.3 — ECS: Multi-AZ Task Placement

Currently Fargate tasks may all land in the same AZ. If that AZ has issues, no browser sessions can start.

Implementation: In CDK task definition, ensure subnets span 2+ AZs. Add placementStrategies: [{ type: 'spread', field: 'attribute:ecs.availability-zone' }].
Trade-off: None — no additional cost.
CAP Impact: Improves Availability of browser test sessions.

1.4 — Lambda: Reserved Concurrency + Provisioned

Lambda cold starts add 1-3s latency. Under burst load, all instances may be cold simultaneously.

Implementation: Set reservedConcurrency: 5 for reader/writer Lambdas. Consider provisionedConcurrency: 1 for the reader to eliminate cold starts.
Trade-off: Provisioned concurrency: ~$15/mo. Reserved: free.
CAP Impact: Improves Availability under load.

1.5 — CloudWatch Alarms + SNS Notifications

No alerting today. Team doesn't know if Aurora is paused too long, Lambda errors spike, or ECS tasks fail to start.

Implementation: Create CloudWatch alarms for: Aurora connection errors, Lambda error rate > 5%, ECS task launch failures, Bedrock throttling. Route to SNS → email/Slack.
Trade-off: ~$1/mo for alarms. Requires Slack webhook or email setup.
CAP Impact: Doesn't change CAP directly but enables fast detection of availability issues.
2 Tier 2 — Production Hardening 1–2 weeks effort

2.1 — API Gateway in Front of Lambda

Lambda functions are currently invoked directly from the browser via AWS SDK. No rate limiting, no request validation, no caching, no throttling.

Implementation: Create API Gateway REST API with /test-cases, /run-records endpoints. Add usage plans, throttling (100 req/s), and Cognito authorizer. Update React app to call API GW instead of Lambda directly.
Trade-off: Adds ~50ms latency per request. ~$3.50/million requests.
CAP Impact: Improves Availability via throttling (prevents overload) and Consistency via request validation.

2.2 — Dead Letter Queues for Lambda

If a Lambda invocation fails (e.g., Aurora timeout during auto-pause resume), the request is lost silently. Test results may not be saved.

Implementation: Add SQS DLQ to both writer and reader Lambdas. Create a CloudWatch alarm when DLQ depth > 0. Add a retry Lambda that processes DLQ messages.
Trade-off: Minimal cost (~$0.40/million SQS requests).
CAP Impact: Improves Consistency — ensures writes are never lost, even if first attempt fails.

2.3 — WAF on CloudFront

No DDoS protection or rate limiting at the edge. A bot or attack could overwhelm backend services.

Implementation: Attach AWS WAF to CloudFront distribution. Add managed rule groups: AWSManagedRulesCommonRuleSet, AWSManagedRulesKnownBadInputsRuleSet. Add rate-based rule (2000 req/5min per IP).
Trade-off: ~$5/mo for WAF + $0.60/million requests inspected.
CAP Impact: Protects Availability from DDoS and abuse.

2.4 — Secrets Manager Auto-Rotation

Aurora database credentials are stored in Secrets Manager but never rotated. Compromised credentials = permanent access.

Implementation: Enable automatic rotation every 30 days using the AWS-provided rotation Lambda. Ensure RDS Data API picks up rotated secrets automatically.
Trade-off: ~$0.40/secret/month. Rotation Lambda runs monthly.
CAP Impact: Security improvement that prevents Availability loss from credential compromise.

2.5 — Aurora Auto-Pause Retry Pattern

Aurora auto-pause saves cost but causes 25-30s cold start on resume. First request after pause often times out.

Implementation: Add exponential retry with backoff in Lambda functions: retry up to 3 times with 5s/10s/20s delays when Communications link failure error is detected. Add a CloudWatch synthetic canary that pings Aurora every 4 minutes to keep it warm during business hours.
Trade-off: Canary: ~$3/mo. Keeps Aurora warm 8am-6pm, auto-pauses overnight.
CAP Impact: Dramatically improves Availability during business hours. Eliminates the most common user-facing error.

2.6 — S3 Cross-Region Replication

CodePipeline stores build artifacts in S3 in us-east-1. If that region is down, CI/CD cannot function.

Implementation: Enable S3 Cross-Region Replication (CRR) to a bucket in us-west-2 for the pipeline artifact bucket.
Trade-off: Storage cost for replica (~$0.023/GB/month). One-time setup.
CAP Impact: Enables Partition Tolerance for CI/CD artifacts.
3 Tier 3 — Multi-Region (Partition Tolerance) 2–4 weeks effort

3.1 — DynamoDB Global Tables

Replicate the prompt2test-config table to us-west-2. Both regions can read and write, with sub-second replication.

Implementation: Enable DynamoDB Global Tables for the config table. Add us-west-2 as a replica region. All writes automatically replicate.
Trade-off: 1.5x write cost (replicated writes charged). ~1s replication lag.
CAP Impact: Full AP — Config is available in both regions. Eventual consistency between regions (~1s). Fully partition tolerant.

3.2 — Aurora Global Database

Add a read replica cluster in us-west-2. Primary in us-east-1 handles writes, us-west-2 handles reads with ~1s lag. Can promote to writer in <1 minute during failover.

Implementation: Create Aurora Global Database. Add secondary cluster in us-west-2. Update reader Lambda to prefer local region's cluster. Set up failover runbook for promotion.
Trade-off: ~$40-60/mo for secondary cluster (min 0.5 ACU). Cross-region data transfer costs.
CAP Impact: CP with HA — Writes are strongly consistent (single writer). Reads from secondary are eventually consistent (~1s). Partition tolerant with manual failover.

3.3 — ECS Cluster in us-west-2

Deploy a second ECS Fargate cluster in us-west-2 with identical task definitions. Browser sessions can run in either region.

Implementation: CDK multi-region stack. Deploy ECS cluster, ECR replicas, and security groups in us-west-2. Route browser session requests to nearest region.
Trade-off: ECR replication: ~$0.10/GB. No idle cost (on-demand tasks). Double deployment pipeline needed.
CAP Impact: Full AP — Browser sessions available from either region. Partition tolerant.

3.4 — Route 53 Health Checks + DNS Failover

Use Route 53 to automatically route traffic to us-west-2 when us-east-1 health checks fail.

Implementation: Create Route 53 health checks against the API endpoint in each region. Configure failover routing policy: primary = us-east-1, secondary = us-west-2. TTL = 60s for fast failover.
Trade-off: ~$1/health check/month. 60-90s failover time (DNS propagation).
CAP Impact: Automatic Partition Tolerance at the DNS layer. Users are transparently routed to healthy region.

3.5 — Cognito Backup & Custom Sync

Cognito has no native multi-region support. A us-east-1 outage means no one can log in.

Implementation: Option A: Pre-issue long-lived JWT refresh tokens (24h) so existing sessions survive region failure. Option B: Create a backup User Pool in us-west-2 with Lambda-based user sync (triggered by Cognito events). Option C: Move to a multi-region auth provider (Auth0, Okta).
Trade-off: Option A is simplest (1 day). Option B is ~1 week. Option C requires migration.
CAP Impact: Improves Availability of authentication during region failure. Option A provides graceful degradation; Option B provides full failover.

3.6 — Bedrock Cross-Region Fallback

If Bedrock in us-east-1 is throttled or unavailable, test plan generation fails entirely.

Implementation: In the AgentCore container, add a fallback: if Bedrock us-east-1 returns 429 or 5xx, retry against us-west-2 Bedrock endpoint. Use boto3 client with configurable region.
Trade-off: Cross-region API call adds ~50ms latency. Same pricing.
CAP Impact: AP — AI layer remains available even during regional Bedrock issues. Partition tolerant.
4 Tier 4 — Active-Active & Platform Maturity 1–2 months effort

4.1 — AWS Global Accelerator

Replace CloudFront-only routing with Global Accelerator for API traffic. Provides anycast IPs and automatic failover at the TCP layer (faster than DNS failover).

Implementation: Create Global Accelerator with endpoints in both regions. Health checks determine routing. Failover in <30s vs 60-90s for DNS.
CAP Impact: Maximum Availability + Partition Tolerance with near-instant failover.

4.2 — Multi-Region CI/CD

Deploy to both regions simultaneously. A single git push triggers parallel builds and deployments to us-east-1 and us-west-2.

Implementation: CodePipeline with parallel deploy stages, or use CDK Pipelines with cross-region support. ECR replication rules for container images.
CAP Impact: Ensures both regions are always running the same version. Consistency of deployments.

4.3 — Multi-Account Strategy

Separate AWS accounts for dev, staging, and production. Blast radius isolation — a bad deploy in dev cannot affect production.

Implementation: AWS Organizations with 3 accounts. CDK deploys to each via cross-account roles. Shared Cognito or per-account pools.
CAP Impact: Isolation improves Availability — production is insulated from dev/test failures.

4.4 — Chaos Engineering

Regularly test failover by intentionally breaking things. Verify that AZ failure, Aurora pause, Lambda throttling, and region failover all work as designed.

Implementation: Use AWS Fault Injection Simulator (FIS) to simulate: AZ impairment, RDS failover, Lambda throttling, ECS task failures. Run monthly in staging.
CAP Impact: Validates that all CAP improvements actually work under real failure conditions.
